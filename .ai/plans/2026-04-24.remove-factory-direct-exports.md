# Plan: Remove Factory Pattern — Direct Exports + Module Augmentation

## Context

The library currently forces consumers through a factory indirection: call `initDialogsValve(dialogs)`, re-export the returned hooks/helpers from a local registry file, and import everything from that registry file. The goal is to eliminate this indirection: consumers pass `dialogs` directly as a prop to `DialogsValveProvider` and import all hooks/helpers straight from the library, with full TypeScript type safety preserved — **no generics at call sites**.

**Approach: Module augmentation** (same pattern as i18next typed translations)
The library exposes an empty `DialogsValveRegistry` interface. The consumer augments it once (in the same file as the `dialogs` definition). All hooks and helpers automatically read their key types from this augmented interface — no `<typeof dialogs>` generic needed at every call site.

---

## New Consumer API

```typescript
// dialogs.ts — define dialogs AND augment the module once
import type { DialogMap } from "@dialogs-valve/react";

export const dialogs = {
  "hello-modal": { Component: HelloModal },
  "user-profile": { Component: UserProfileDrawer },
} as const satisfies DialogMap;

// One-time setup: declare the registry so all hooks/helpers auto-type
declare module "@dialogs-valve/react" {
  interface DialogsValveRegistry {
    dialogs: typeof dialogs;
  }
}

// App.tsx — pass dialogs as prop, import provider directly from library
import { DialogsValveProvider } from "@dialogs-valve/react";
import { dialogs } from "./dialogs"; // this import also applies the augmentation

<DialogsValveProvider dialogs={dialogs} onNavigate={navigate}>
  <App />
</DialogsValveProvider>

// Component.tsx — import hook directly, no generic needed
import { useDialogsValve, buildDialogUrl } from "@dialogs-valve/react";

const { openDialog } = useDialogsValve()!;
openDialog("hello-modal");   // ✓ valid key
openDialog("typo-key");      // ✗ TypeScript error — caught at compile time

const url = buildDialogUrl("user-profile"); // ✓ typed, no generic at call site
```

## Implementation Steps

### 1. `src/types.ts` — Add `DialogsValveRegistry` interface and `RegisteredDialogKeys`

Append at the end of the file:

```typescript
/**
 * Augment this interface in your app to register your dialogs map.
 * All hooks and helpers will automatically use the registered keys.
 *
 * @example
 * declare module "@dialogs-valve/react" {
 *   interface DialogsValveRegistry {
 *     dialogs: typeof import("./my-dialogs").dialogs;
 *   }
 * }
 */
export interface DialogsValveRegistry {}

/**
 * Computed dialog key union from the augmented registry.
 * Falls back to `string` when no registry is declared.
 */
export type RegisteredDialogKeys =
  DialogsValveRegistry extends { dialogs: infer D extends DialogMap }
    ? keyof D & string
    : string;
```

### 2. `src/hooks.ts` — Default generic to `RegisteredDialogKeys`

Change the generic default from `string` to `RegisteredDialogKeys`:

```typescript
import type { RegisteredDialogKeys } from "./types";

export function useDialogsValve<TKeys extends string = RegisteredDialogKeys>():
  DialogsValveContextValue<TKeys> | null {
  const context = useContext(DialogsValveContext);
  // ...
  return context as DialogsValveContextValue<TKeys> | null;
}
```

The `as` cast is needed because the context is stored as `DialogsValveContextValue<string>` at runtime. This is the same compile-time narrowing the factory does today.

### 3. `src/services.ts` — Add generics defaulting to `RegisteredDialogKeys`

| Function | Change |
|---|---|
| `buildDialogUrl` | `(dialogKey: string, ...)` → `<TKeys extends string = RegisteredDialogKeys>(dialogKey: TKeys, ...)` |
| `buildCloseDialogUrl` | same pattern |
| `extractDialogProps` | same pattern |
| `cleanUpQueryParams` | same pattern |
| `getActiveDialogKeys` | already generic — change default from `string` to `RegisteredDialogKeys` |
| `validateDialogKeys` | already generic — change default same way |

| `buildCloseAllDialogsUrl` | no key param — no change |
